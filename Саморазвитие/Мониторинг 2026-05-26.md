### Ошибка: launchd — скрипт не найден / Job failed to exec

**Симптомы:** daemon не запускается, в логах `Job failed to exec(3)` или тишина

**Причина:** launchd не наследует PATH из shell. Относительные пути не работают.

**Решение:**
```bash
# В .plist — ТОЛЬКО абсолютные пути:
<key>ProgramArguments</key>
<array>
    <string>/bin/bash</string>
    <string>/Users/rodion/.claude/agents/local-agent.sh</string>
</array>

# Проверить исполняемость:
chmod +x /Users/rodion/.claude/agents/local-agent.sh

# Перезагрузить агент:
launchctl unload ~/Library/LaunchAgents/com.rodion.local-agent.plist
launchctl load -w ~/Library/LaunchAgents/com.rodion.local-agent.plist
```

---

### Ошибка: launchd — Operation not permitted

**Симптомы:** `launchctl load` даёт ошибку "operation not permitted"

**Причина:** попытка загрузить LaunchDaemon (system-level) без sudo, или защита SIP

**Решение:**
```bash
# Для LaunchDaemons (system) — нужен sudo:
sudo launchctl load -w /Library/LaunchDaemons/com.name.plist

# Для LaunchAgents (пользователь) — без sudo:
launchctl load -w ~/Library/LaunchAgents/com.name.plist
```

---

### Ошибка: launchd — логи пустые, скрипт не пишет вывод

**Симптомы:** агент "работает" по launchctl list, но ничего не происходит

**Причина:** нет StandardOutPath / StandardErrorPath в plist

**Решение:**
```xml
<key>StandardOutPath</key>
<string>/Users/rodion/.claude/agents/my-agent.log</string>
<key>StandardErrorPath</key>
<string>/Users/rodion/.claude/agents/my-agent-error.log</string>
```
> ⚠️ НЕ указывать пути в ~/Desktop, ~/Documents, ~/Downloads — macOS блокирует запись из launchd без TCC разрешения!

---

### Ошибка: launchd — скрипт не запускается из-за прав на файл

**Симптомы:** plist корректный, но скрипт молчит

**Причина:** неправильный owner или отсутствие executable bit

**Решение:**
```bash
# Для LaunchDaemon (system-level):
sudo chown root:wheel /path/to/script.sh
sudo chmod +x /path/to/script.sh

# Для LaunchAgent (user-level):
chown $USER /path/to/script.sh
chmod +x /path/to/script.sh
```

### Ошибка: Telegram бот падает при ошибке сети

**Симптомы:** бот умирает после `telegram.error.NetworkError` или `TimedOut`

**Причина:** отсутствует глобальный error_handler — исключение не перехватывается

**Решение:**
```python
import logging
import traceback
from telegram import Update
from telegram.ext import Application, ContextTypes

# Настройка логирования (убрать шум httpx)
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s - %(name)s - %(levelname)s - %(message)s'
)
logging.getLogger('httpx').setLevel(logging.WARNING)
logging.getLogger('httpcore').setLevel(logging.WARNING)

# Глобальный обработчик ошибок
async def error_handler(update: object, context: ContextTypes.DEFAULT_TYPE):
    tb = "".join(traceback.format_exception(
        None, context.error, context.error.__traceback__
    ))
    logging.error(f"Ошибка: {context.error}\n{tb}")
    
    # Уведомить в Telegram
    if update and hasattr(update, 'effective_message'):
        await update.effective_message.reply_text("⚠️ Произошла ошибка")

app = Application.builder().token(TOKEN).build()
app.add_error_handler(error_handler)
```

> ✅ С add_error_handler бот продолжает работу после любой ошибки!

### Ошибка: Telegram бот — пользователь заблокировал бота, Forbidden

**Симптомы:** `telegram.error.Forbidden: Forbidden: bot was blocked by the user`

**Причина:** попытка отправить сообщение пользователю, который заблокировал бота

**Решение:**
```python
from telegram.error import Forbidden

async def send_safe(bot, chat_id, text):
    try:
        await bot.send_message(chat_id=chat_id, text=text)
    except Forbidden:
        logging.warning(f"Пользователь {chat_id} заблокировал бота")
        # можно удалить из БД или пометить как inactive
```

### Ошибка: Obsidian — заметки исчезают сразу после создания

**Симптомы:** создаёшь заметку, она пропадает через несколько минут

**Причина:** race condition: remote версия перезаписывает локальную (созданную офлайн)

**Решение:**
1. Открыть **Settings → File recovery**
2. Найти заметку по времени создания
3. Восстановить

Для предотвращения: перед созданием важных заметок убедиться, что sync завершён (зелёная иконка).

---

### Ошибка: Obsidian — настройки/плагины не синхронизируются между устройствами

**Симптомы:** плагин включён на Mac, на iPhone отключён; горячие клавиши разные

**Причина:** Obsidian не поддерживает live-reload настроек — требуется перезапуск

**Решение:**
```
1. На Mac: дождаться завершения sync (зелёная иконка)
2. На iPhone: force-quit Obsidian → открыть заново
3. Настроить стратегию конфликтов: Settings → Sync → Conflict Strategy
```

**Лучшая стратегия:** Settings → Sync → "Create conflict file" (для ручного разрешения важных конфликтов)

### Ошибка: Obsidian — конфликтные файлы "(Conflicted copy)"

**Симптомы:** появляются файлы `Заметка (Conflicted copy MacBook 20261205).md`

**Причина:** одновременное редактирование на нескольких устройствах до завершения sync

**Решение:**
1. Оба файла открыть рядом
2. Скопировать нужный контент в основной
3. Удалить `(Conflicted copy)` файл
4. Использовать Sync log (Settings → Sync → View activity) чтобы понять хронологию

**Алгоритм Obsidian:** markdown → merge (как git), остальные файлы → last-modified-wins.

### Ошибка: GitHub Actions — workflow висит, тратит минуты

**Симптомы:** job не завершается, статус "running" часами, деньги утекают

**Причина:** нет явного timeout, дефолт — 6 часов

**Решение:**
```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 10   # ← ОБЯЗАТЕЛЬНО для каждого job
    steps:
      - uses: actions/checkout@v4
```

### Ошибка: GitHub Actions — зависимости устанавливаются каждый раз (медленно)

**Симптомы:** `npm install` или `pip install` занимает 2-3 мин при каждом run

**Причина:** отсутствует кэширование

**Решение:**
```yaml
# Для Node.js:
- uses: actions/cache@v4
  with:
    path: |
      ~/.npm
      node_modules
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-

# Для Python:
- uses: actions/cache@v4
  with:
    path: ~/.cache/pip
    key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }}
```

> 💡 Правильный кэш сокращает время workflow на ~60%

```
Сначала составь план в виде нумерованного списка.
Не начинай кодировать пока план не утверждён (напиши "PLAN READY").
```

### 3. Telegram бот — структурированное логирование

Добавить в `~/Desktop/qsnera-reels-bot/bot.py`:
```python
# В начале файла, после импортов:
logging.getLogger('httpx').setLevel(logging.WARNING)
logging.getLogger('httpcore').setLevel(logging.WARNING)
logging.getLogger('apscheduler').setLevel(logging.WARNING)
```
Это уберёт 80% лишнего шума в Railway logs.

### 4. GitHub Actions — добавить timeout к каждому job

В `.github/workflows/ai-agent.yml` и `memory-agent.yml`:
```yaml
jobs:
  run:
    timeout-minutes: 15   # ← добавить везде
```

### 5. Контекстная инженерия (Context Engineering)

Для промптов в GitHub Actions задачах:
- **JIT загрузка:** вместо "дай мне все файлы клиента" → "найди клиента {id} и прочитай только нужные поля"
- **Compaction:** для длинных задач добавлять "Кратко резюмируй промежуточные результаты каждые N шагов"
- **Четыре столпа:** System Prompt (краткий) + Tools (конкретные) + Data (JIT) + Sub-agents (для длинных задач)

### 6. Obsidian — настроить стратегию конфликтов

```
Settings → Sync → Conflict Strategy → "Create conflict file"
```
Настроить на **каждом** устройстве отдельно (Mac + iPhone).
